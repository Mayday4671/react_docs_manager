# 开发工作流规范

## 功能变更文档记录（强制执行）

### 规则
每次完成功能开发、修复或优化后，**必须**在 `docs/` 目录下记录变更内容，不得遗漏。

### 目录结构
```
docs/
└── YYYY-MM-DD/          ← 按日期分目录，格式为当天日期
    └── 功能名称.md       ← 每个独立功能/修复单独一个文件
```

### 文档格式模板
```markdown
# 功能名称

## 完成时间
YYYY-MM-DD

## 功能概述
简要描述本次变更的目的和内容。

## 主要功能
- ✅ 功能点1
- ✅ 功能点2

## 相关文件
### 前端
- `src/frontend/components/xxx.tsx` — 说明

### 后端
- `src/app/api/xxx/route.ts` — 说明
- `src/backend/services/xxx.ts` — 说明

## 注意事项
（可选）需要特别注意的事项、已知问题等
```

### 执行时机
以下情况**必须**写文档：
1. 新增功能模块
2. 修复 Bug（说明问题原因和修复方式）
3. 重构或优化（说明改动范围）
4. 新增依赖包
5. 数据库结构变更

### 执行时机（不需要写）
- 单纯补全注释
- 修改样式细节
- 修复明显的笔误

---

## 代码提交规范

### 提交时机
功能完成并验证无误后提交，不要攒太多改动一起提交。

### commit message 格式
```
<type>(<scope>): <subject>

<body>（可选，多行说明）
```

**type 类型：**
- `feat` — 新功能
- `fix` — Bug 修复
- `docs` — 文档/注释
- `refactor` — 重构
- `chore` — 构建/依赖/配置

**示例：**
```
feat(菜单管理): 新增菜单管理页面

- 树形表格展示所有菜单
- Key 自动生成并校验唯一性
- 可视化图标选择器
```

### 推送规则
- 始终推送到 `main` 分支（当前项目为个人项目，无需 PR 流程）
- 每次提交后立即推送，不要积压本地提交
